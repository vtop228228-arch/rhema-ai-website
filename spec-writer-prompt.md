# SPEC_WRITER_PROMPT — Промпт для генерации спецификации

> **Тип:** Инструкция (Instruction)
> **Назначение:** Загрузи этот промпт + PROJECT_IDEA.md в Claude AI (web) — получи полную техническую спецификацию SPEC.md
> **Формат выхода:** Markdown-документ, готовый к передаче в Claude Code

---

## Философия этого документа

Этот файл — **готовый промпт для Claude AI**. Вы не пишете его сами — копируете и используете.

**Принцип работы:**
- Вы — архитектор и коммуникатор
- Claude — исполнитель: читает промпт, делает работу
- Ваша задача — описать контекст и скопировать правильный промпт

**Workflow:**
```
Вы → копируете промпт → загружаете в Claude AI (web)
       ↓
       прикрепляете свой контекст (PROJECT_IDEA.md)
       ↓
Claude → выполняет задачу по промпту
       ↓
Вы → получаете результат (SPEC.md) → используете в Claude Code
```

Промпт уже отлажен. Не нужно его улучшать с первого раза — просто используйте.

# Критические правила

## Аудитория спецификации — AI-агенты, НЕ люди

Спецификация будет читаться AI-агентами (Claude Code, субагенты).
Это значит:

1. МАКСИМАЛЬНАЯ КОНКРЕТНОСТЬ — никаких абстракций, никаких "стандартный подход"
2. ПОЛНЫЕ ПРИМЕРЫ — вместо "формат ответа JSON" покажи реальный JSON с данными
3. ГОТОВЫЙ КОД — SQL-миграции, API-хендлеры, Zod-схемы должны быть готовы к copy-paste
4. ЗАПРЕТ НА TODO — каждое решение должно быть принято СЕЙЧАС
5. ЗАПРЕТ НА PLACEHOLDER — вместо "your-api-key" пиши "process.env.API_KEY"
6. ЗАПРЕТ НА ВОПРОСЫ — не пиши "нужно уточнить у клиента". Если данных нет — прими решение и обоснуй
7. ЗАПРЕТ НА "ОЧЕВИДНО" — AI-агент не знает, что очевидно. Опиши каждый шаг

# 6 обязательных блоков

Спецификация ОБЯЗАНА содержать ровно 6 блоков в следующем порядке.

## БЛОК 0: Обзор проекта

Напиши:
- Что это (1-2 предложения)
- Полный стек с версиями
- Таблица ролей пользователей (роль, описание, доступ)
- Структура URL-маршрутов приложения

## БЛОК 1: User Stories

Для каждой ключевой функции из PROJECT_IDEA.md:
- Формат: "Как [персонаж], я хочу [действие], чтобы [результат]"
- Пошаговый сценарий (5-8 шагов)
- Критерии приёмки (чеклист)

Требования:
- Персонаж — конкретная роль, НЕ "пользователь"
- Каждый сценарий включает и happy path, и что происходит при ошибке
- Минимум 5 user stories для MVP

## БЛОК 2: Data Model

Для КАЖДОЙ таблицы:
- Полный CREATE TABLE с типами, constraints, defaults
- Все CHECK constraints для enum-полей
- Индексы на поля с фильтрацией/сортировкой
- RLS-политики (SELECT, INSERT, UPDATE, DELETE) для каждой роли
- Триггер moddatetime для updated_at
- Комментарии, объясняющие нетривиальные решения

Требования:
- SQL готов к выполнению в Supabase SQL Editor
- Денежные значения — INTEGER (копейки)
- UUID для всех id
- Каскадное удаление или SET NULL — указать явно для каждого FK
- Диаграмма связей в ASCII-формате

## БЛОК 3: API Endpoints

Для КАЖДОГО эндпоинта:
- Метод + путь (Next.js App Router: /api/[resource]/route.ts)
- Параметры запроса с типами
- Полный JSON тела запроса (с реалистичными данными)
- Полный JSON ответа 200 (с реалистичными данными)
- Полный JSON ответа ошибки (400, 401, 404, 500)
- Zod-схема валидации

Требования:
- Каждый ответ содержит реалистичные данные, а не "string" или "..."
- Пагинация: { data: [], meta: { total, page, per_page } }
- Ошибки: { error: { code: "ERROR_CODE", message: "Описание" } }

## БЛОК 4: UI/UX

Для КАЖДОГО экрана:
- URL-путь
- Layout (какой layout используется)
- Список компонентов с описанием
- Три обязательных состояния: Loading (skeleton/spinner), Empty (CTA), Error (toast/inline)
- Все действия пользователя на экране и что происходит
- Responsive-поведение (desktop / tablet / mobile)

Требования:
- Компоненты shadcn/ui: указать конкретные (Button, Card, Table, Sheet, Dialog)
- Иконки: Lucide React (указать конкретные иконки)
- Цветовая схема: нейтральная (не выдумывай бренд)

## БЛОК 5: Business Logic

Описать:
- Правила валидации для каждой формы (поле, тип, min/max, regex)
- Бизнес-правила (лимиты, ограничения, условия)
- Процесс аутентификации пошагово (регистрация, вход, восстановление пароля)
- Все внешние интеграции (для каждой: что отправляем, что получаем, retry-стратегия)
- Безопасность: CORS, rate limiting, input sanitization
- Cron-задачи (если есть): расписание, что делает, обработка ошибок

Требования:
- Для каждого правила — что происходит при нарушении
- Для каждой интеграции — fallback при недоступности сервиса

## БЛОК 6: Edge Cases

Категории:
1. Сеть: потеря соединения, таймауты, медленный интернет
2. Данные: пустые состояния, конкурентное редактирование, невалидные данные
3. Безопасность: XSS, SQL injection (Supabase защищает), подмена ID
4. Лимиты: большие файлы, длинные тексты, 10K+ записей
5. Платежи (если есть): двойная оплата, отмена, возврат, webhook не дошёл
6. Время: часовые пояса, переход на летнее время, дедлайны в полночь

Требования:
- Минимум 15 edge cases
- Для каждого: ситуация, триггер, ожидаемое поведение системы

## БЛОК 1: User Stories

### US-001: Регистрация нового пользователя
**Как** фрилансер, впервые зашедший на TaskFlow,
**я хочу** зарегистрироваться по email,
**чтобы** начать управлять задачами.

**Сценарий:**
1. Пользователь заходит на taskflow.ru
2. Нажимает "Попробовать бесплатно"
3. Перенаправляется на /auth/register
4. Вводит: имя, email, пароль (мин. 8 символов)
5. Нажимает "Создать аккаунт"
6. Supabase Auth отправляет email подтверждения
7. Пользователь кликает по ссылке в письме
8. Перенаправляется на /dashboard с пустым состоянием
9. Видит онбординг: "Создайте свой первый проект"

**Критерии приёмки:**
- [ ] Пароль: минимум 8 символов, хотя бы 1 цифра
- [ ] Дубликат email: ошибка "Этот email уже зарегистрирован"
- [ ] Письмо подтверждения приходит в течение 30 секунд
- [ ] После подтверждения — автоматический вход без повторного ввода пароля
- [ ] При попытке войти без подтверждения: "Подтвердите email. Отправить письмо повторно?"

### US-002: Создание проекта
**Как** фрилансер с новым клиентом,
**я хочу** создать проект за 15 секунд,
**чтобы** сразу начать добавлять задачи.

**Сценарий:**
1. Пользователь на /dashboard нажимает "+ Новый проект"
2. Открывается Dialog с формой
3. Вводит: название проекта (обязательно), имя клиента (опционально),
   цвет маркера (выбор из 8 цветов), почасовая ставка (опционально)
4. Нажимает "Создать"
5. Проект появляется в sidebar
6. Автоматически перенаправляется на /projects/[id]
7. Видит пустую канбан-доску с подсказкой

**Критерии приёмки:**
- [ ] Название проекта: 1-100 символов
- [ ] Бесплатный план: максимум 3 активных проекта
- [ ] При попытке создать 4-й: модалка с предложением подписки
- [ ] Проект появляется в sidebar мгновенно (optimistic update)

## БЛОК 2: Data Model

### Диаграмма связей
```
auth.users 1──N projects (user_id)
projects   1──N tasks (project_id)
tasks      1──N time_entries (task_id)
auth.users 1──N time_entries (user_id)
auth.users 1──N chat_messages (user_id)
auth.users 1──1 user_settings (user_id)
```

### profiles
```sql
-- Расширение auth.users — публичная информация пользователя
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  full_name TEXT NOT NULL,
  avatar_url TEXT,
  telegram_chat_id BIGINT, -- для отправки уведомлений
  plan TEXT NOT NULL DEFAULT 'free' CHECK (plan IN ('free', 'pro')),
  daily_chat_messages_used INTEGER NOT NULL DEFAULT 0,
  daily_chat_reset_at TIMESTAMPTZ DEFAULT NOW(),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

CREATE POLICY "profiles_select_own"
  ON profiles FOR SELECT
  USING (auth.uid() = id);

CREATE POLICY "profiles_update_own"
  ON profiles FOR UPDATE
  USING (auth.uid() = id);

-- Автосоздание профиля при регистрации
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO profiles (id, full_name)
  VALUES (NEW.id, COALESCE(NEW.raw_user_meta_data->>'full_name', 'Пользователь'));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW
  EXECUTE FUNCTION handle_new_user();
```

## Типичные ошибки при генерации

Если Claude AI сгенерировал спецификацию с этими проблемами — исправь вручную или попроси переделать:

| Ошибка | Как выглядит | Как исправить |
|--------|-------------|---------------|
| Абстрактные описания | "Используется стандартная аутентификация" | Запроси: "Опиши процесс регистрации пошагово с кодом" |
| Пропущены RLS | Таблицы без CREATE POLICY | Запроси: "Добавь RLS-политики для каждой таблицы" |
| Нет примеров ответов API | "Возвращает JSON с данными" | Запроси: "Покажи полный JSON-ответ с реалистичными данными" |
| Нет состояния Empty в UI | Экран описан только с данными | Запроси: "Добавь Empty-состояние для каждого экрана" |
| TODO в тексте | "TODO: определить формат позже" | Запроси: "Прими решение и зафиксируй в спецификации" |
| Edge cases < 10 | Только 3-4 очевидных сценария | Запроси: "Добавь edge cases для сети, безопасности, лимитов, платежей" |
| Stripe вместо ЮKassa | "Подключить Stripe для платежей" | Замени на ЮKassa — Stripe не работает в России |
| Edge Functions | "Создать Supabase Edge Function" | Замени на API Route в Next.js или Node.js на VPS |
