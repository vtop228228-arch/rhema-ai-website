## Философия этого документа

**Вы не пишете SPEC.md вручную.** Claude AI генерирует её из вашего PROJECT_IDEA.md по промпту из SPEC_WRITER_PROMPT.md.

Зачем тогда этот справочник?

1. **Валидация** — понять, все ли 6 блоков Claude заполнил правильно
2. **Корректировка** — если есть дыры (нет RLS, нет примеров API, слабые edge cases) — знать как попросить доработать
3. **Коммуникация** — говорить с Claude на языке "добавь RLS-политики в Блок 2" вместо "это не полная"
4. **Контроль качества** — убедиться, что SPEC.md пригодна для автономной сборки Claude Code

**Правильный workflow:**
```
Вы → PROJECT_IDEA.md (уже валидирован)
Claude AI → генерирует SPEC.md по SPEC_WRITER_PROMPT
Вы → используете этот справочник чтобы валидировать результат
Claude → дорабатывает блоки по вашим замечаниям
Claude Code → автономно строит продукт по SPEC.md
```

## Зачем нужна спецификация

SPEC.md — это **главный документ проекта**. Claude Code читает его и строит продукт автономно.

Без качественной спецификации Claude Code:
- Задаёт уточняющие вопросы (теряется время)
- Принимает архитектурные решения самостоятельно (теряется контроль)
- Генерирует placeholder-код (теряется качество)

С хорошей спецификацией Claude Code:
- Создаёт полную базу данных за один промпт
- Строит API, которое работает с первого раза
- Генерирует UI, соответствующий бизнес-логике

**Критическое правило:** Спецификация пишется **ДЛЯ AI-АГЕНТОВ**, не для людей. Claude должен сгенерировать её с учётом этого:
- Максимальная конкретность (никаких абстракций)
- Полные примеры вместо описаний
- НЕТ TODO и placeholder
- НЕТ формулировок "определить позже" или "на усмотрение разработчика"

Если Claude нарушил хоть одно правило — попросите переписать конкретный блок.

## БЛОК 2: Data Model

### Правила написания
- Полный SQL — готов к копированию в Supabase SQL Editor
- Все таблицы с id (UUID), created_at, updated_at
- Все foreign keys с ON DELETE CASCADE или SET NULL (указать явно)
- Индексы на все поля, по которым будет поиск или фильтрация
- RLS-политики для КАЖДОЙ таблицы
- Триггеры для updated_at

### Таблица: [название]
```sql
CREATE TABLE [название] (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  -- поля с типами и constraints
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_[название]_[поле] ON [название]([поле]);
ALTER TABLE [название] ENABLE ROW LEVEL SECURITY;

CREATE POLICY "[название]_select_own" ON [название] FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "[название]_insert_own" ON [название] FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "[название]_update_own" ON [название] FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY "[название]_delete_own" ON [название] FOR DELETE USING (auth.uid() = user_id);
```

### Связи: `[таблица_1] 1──N [таблица_2] (через FK)` | `[таблица_2] N──M [таблица_3] (через junction)`

> **Что НЕ писать в Data Model:**
> - "Добавить поля по необходимости" — добавь сейчас
> - "Схема может измениться" — зафиксируй текущую версию
> - Таблицы без RLS-политик

## Что Claude вернёт для TaskFlow (фрагменты)

Ниже — фрагменты того, что Claude сгенерирует для TaskFlow, если загрузить его PROJECT_IDEA.md в SPEC_WRITER_PROMPT. Используйте как образец для сравнения с тем, что получите в своём проекте. В реальной спецификации каждый блок значительно подробнее.

### Блок 1: User Stories (фрагмент)

```markdown
### US-001: Быстрое добавление задачи
**Как** фрилансер-дизайнер с 5 проектами,
**я хочу** добавить задачу одним сообщением (текст + дедлайн),
**чтобы** не тратить больше 10 секунд на создание задачи.

**Сценарий:**
1. Пользователь нажимает "+" или Ctrl+N
2. Появляется поле ввода с плейсхолдером "Что нужно сделать?"
3. Пользователь пишет: "Отправить макет логотипа до пятницы"
4. Система парсит "до пятницы" → ставит дедлайн на ближайшую пятницу
5. Пользователь выбирает проект из dropdown (последний используемый — по умолчанию)
6. Enter → задача создана, появляется на канбан-доске

**Критерий приёмки:**
- [ ] Время от клика до создания задачи < 10 секунд
- [ ] Парсинг дедлайна из текста на русском ("завтра", "через 3 дня", "до пятницы")
- [ ] Дефолтный проект = последний используемый
- [ ] Задача мгновенно появляется на доске (optimistic update)
```

### Блок 2: Data Model (фрагмент)

```sql
CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  client_name TEXT,
  color TEXT DEFAULT '#6366F1',
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'archived', 'completed')),
  hourly_rate INTEGER, -- в копейках, NULL = не считаем по часам
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_projects_user_id ON projects(user_id);
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;

CREATE POLICY "projects_select_own" ON projects FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "projects_insert_own" ON projects FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "projects_update_own" ON projects FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY "projects_delete_own" ON projects FOR DELETE USING (auth.uid() = user_id);
```

### Блок 3: API (фрагмент)

```markdown
#### `POST /api/tasks`
**Описание:** Создание задачи с парсингом дедлайна
**Авторизация:** Bearer token

**Запрос:** `{ "title": "Отправить макет до пятницы", "project_id": "a1b2c3d4-...", "priority": "medium" }`
**Ответ 201:** `{ "data": { "id": "f7e6d5c4-...", "title": "...", "status": "todo", "deadline": "2026-04-17T23:59:59Z" } }`
**Ответ 400:** `{ "error": { "code": "VALIDATION_ERROR", "message": "Field 'title' is required" } }`
```

### Блок 4: UI/UX (фрагмент)

```markdown
### Экран: Канбан-доска проекта
**Путь:** `/projects/[id]` | **Layout:** Sidebar + Main

**Компоненты:** ProjectSidebar, KanbanBoard (4 колонки), TaskCard, QuickAddInput, TimeTracker
**Loading:** Skeleton-карточки (3 на колонку) | **Empty:** "Создайте первую задачу" + CTA | **Error:** Toast

**Действия:**
1. Drag & drop TaskCard → PATCH /api/tasks/[id] (status + position)
2. Клик TaskCard → Sheet справа с деталями
3. Enter в QuickAddInput → POST /api/tasks
```

### Блок 6: Edge Cases (фрагмент)

```markdown
| # | Ситуация | Поведение системы |
|---|----------|-------------------|
| 1 | Пропала сеть при drag & drop | Optimistic update + retry при восстановлении |
| 2 | API Supabase недоступен | Toast + exponential backoff (3/6/12 сек) |
| 3 | Telegram Bot API не отвечает | Очередь notification_queue, retry каждые 5 мин |
| 4 | Удаление проекта с задачами | Модалка подтверждения → CASCADE delete |
| 5 | Парсер не распознал дедлайн | Задача без дедлайна + тултип "Добавьте вручную" |
| 6 | Лимит 3 проектов (free) | Модалка с предложением подписки |
```

```

  ↓ (используйте 2-10-PROJECT_IDEA_TEMPLATE.md как reference для валидации)
  
ШАГ 2: Откройте Claude AI (web)
  ↓ Загрузите 3 файла:
  • PROJECT_IDEA.md (ваш)
  • 2-13-SPEC_WRITER_PROMPT.md (из toolkit)
  • 2-11-SPEC_TEMPLATE.md (этот справочник)
  
ШАГ 3: Промпт:
  "Сгенерируй полную техническую спецификацию SPEC.md по инструкциям 
   из SPEC_WRITER_PROMPT и формату из справочника 2-11-SPEC_TEMPLATE.
   Заполни все 6 блоков конкретикой: SQL с RLS, JSON-примеры API, 
   три состояния UI, минимум 10 edge cases."
  
ШАГ 4: Получите результат
  ↓ Claude вернёт SPEC.md на 300-800 строк
  
ШАГ 5: Валидация (этот документ)
  ↓ Пройдитесь по чек-листу выше
  
ШАГ 6: Если нужны правки — попросите Claude
  "В блоке [X] не хватает [Y]. Доработай по формату из справочника."
  
ШАГ 7: Передайте SPEC.md в Claude Code Setup Generator
  ↓ Следующий шаг: генерация CLAUDE.md + субагентов + rules
```

**Выход Setup Generator (следующий шаг пайплайна):**
- CLAUDE.md (max 120 строк)
- Субагенты (.claude/agents/*.md)
- Rules (.claude/rules/*.md)
- Skills (.claude/skills/*/SKILL.md)
- MCP-команды
- Финальный промпт для Claude Code

**Итог:** вы не написали SPEC.md руками. Вы подготовили PROJECT_IDEA.md, запустили генерацию, валидировали результат, дали правки. Это и есть роль архитектора.
